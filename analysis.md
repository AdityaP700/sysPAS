# RunbookMind: Mock vs Real Data — Full Pipeline Analysis

> **Verdict: The project is ~85% mocked.** The architecture, abstractions, and wiring are production-grade, but the actual integrations to Splunk/MCP/AI are not connected. No `.env` file exists, no real credentials, no live Splunk instance.

---

## The Big Picture: Dual-Layer Architecture

The codebase is cleverly structured with an **abstraction layer that separates mock from real**. Every critical subsystem has:
- A `Base*` abstract class (the interface contract)
- A `Mock*` implementation (currently wired in)
- A `SplunkMCP*` implementation (the real one, exists but disconnected)

This is actually a **strength** — the swap path to real is already engineered. You just need credentials and connectivity.

## Layer-by-Layer Breakdown

### Layer 1: SPL Generation (MOCKED)

| Component | File | Status | What it does |
|---|---|---|---|
| `MockGenerator` | `app/spl/generator.py` | ❌ MOCK | Calls `TemplateGenerationEngine` — pattern matching on keywords, fills in hardcoded SPL templates |
| `SplunkMCPGenerator` | `app/splunk/adapters/mcp_generator.py` | ✅ REAL (wiring missing) | Calls `splunk_generate_spl` MCP tool — the real Splunk AI model |
| `MockOptimizer` | `app/spl/optimizer.py` | ❌ MOCK | Literally appends `earliest=-15m` to any SPL string |
| `SplunkMCPOptimizer` | `app/splunk/adapters/mcp_optimizer.py` | ✅ REAL (wiring missing) | Calls `splunk_optimize_spl` MCP tool |
| `MockExplainer` | `app/spl/explainer.py` | ❌ MOCK | Returns `"This mock query explains: '{spl}'"` |
| `SplunkMCPExplainer` | `app/splunk/adapters/mcp_explainer.py` | ✅ REAL (wiring missing) | Calls `splunk_explain_spl` MCP tool |

**The generation engine** (`app/generation/engine.py`) does template-based SPL:
- Maps NL descriptions → `Intent` enum (e.g., `FAILED_LOGIN`, `BRUTE_FORCE`)
- Fills in hardcoded templates with field names like `src_ip`, `user`, `threshold=100`
- No LLM call at all — pure string interpolation

**Key finding:** The real `SplunkMCP*` adapters are fully written (`mcp_generator.py`, `mcp_optimizer.py`, `mcp_explainer.py`, `mcp_validator.py`) but the `RunbookCompiler` in `compiler.py` is injected with Mock implementations via `dependencies.py`.

### Layer 2: Schema Discovery (MOCKED)

| Component | File | Status |

| `SchemaDiscoveryEngine` | `app/schema/discovery.py` | ✅ REAL (wiring missing) — calls `splunk_get_fields` + `splunk_get_indexes` via MCP |

The `MockSchemaProvider` returns:

```python
  "sales_metrics": ["store_id", "revenue_drop_pct", "date"]

```
These are **static, hardcoded dictionaries** — not pulled from any real Splunk instance.

### Layer 3: MCP Client (REAL CODE, NO CONNECTION)

The MCP client (`app/splunk/adapters/client.py`) is **fully implemented** and supports:
- `stdio` transport (subprocess, for local Splunk MCP server)
- `sse` transport (HTTP SSE endpoint, for cloud/remote)

But the settings in `app/config/settings.py` show:
```python
mcp_transport: str = "stdio"
mcp_command: str = "python"
mcp_args: List[str] = ["-m", "splunk_mcp_server"]
enable_mcp: bool = True
```

**There is NO `.env` file**. The `Settings` class uses `env_prefix="RUNBOOKMIND_"` — so env vars like `RUNBOOKMIND_MCP_TRANSPORT`, `RUNBOOKMIND_MCP_SSE_URL` would be read if they existed. Since they don't, defaults kick in — pointing at `python -m splunk_mcp_server`, which is not installed/running.

The gate is this check in `client.py`:
```python
if not settings.enable_mcp:
    raise SplunkMCPAdapterError("MCP adapter execution is disabled in settings.")
```
`enable_mcp=True` by default, but MCP server is not running → all real MCP calls would raise `MCPConnectionError`.

### Layer 5: Connectors — Slack, Jira, ServiceNow, PagerDuty, Teams (DUAL MODE)

These are the most nuanced. Each connector has **real HTTP implementation** inside it, BUT they detect mock tokens and short-circuit:

```python
# From slack.py
if token.startswith("mock-") or token == "valid_token":
    return {"message_id": f"slack_msg_{hash(text) % 100000}", "status": "delivered_mock"}

# Otherwise → real urllib call to https://slack.com/api/chat.postMessage
```

So connectors will work **for real** if you put real tokens in the database. Right now tests use `mock-` prefixed tokens.

```python
sqlite_db_path: str = "runbookmind.db"
```

`runbookmind.db` actually **exists** (1.5 MB on disk). This is a real SQLite database. Storage, tenants, audit logs, bundle store, trace store — these are all fully functional and real. **This layer is not mocked.**

## Where Is the Dependency Injection?

The critical file is `app/web/dependencies.py`. It's the **wiring hub**. Currently it wires:

```python
_query_runner_instance = MockQueryRunner()   # ← SWAP THIS
```

And `RunbookService` receives:
```python
_service_instance = RunbookService(
    repo=_repo_instance,
    bundle_store=_bundle_store_instance,
    compilation_store=_compilation_store_instance,
    trace_store=_trace_store_instance,
)
```

The `RunbookCompiler` gets its adapters from `RunbookService`. **The gap:** the compiler instantiation inside `RunbookService` is where Mock adapters are hardcoded.

## What You Need to Go Real — Step by Step

### Step 1: Create `.env` file

```bash
# MCP Transport — choose stdio (local) or sse (cloud)
RUNBOOKMIND_MCP_TRANSPORT=sse
RUNBOOKMIND_MCP_SSE_URL=https://your-splunk-host:8089/api/mcp

# OR for stdio (local Splunk MCP server):
RUNBOOKMIND_MCP_TRANSPORT=stdio
RUNBOOKMIND_MCP_COMMAND=python
RUNBOOKMIND_MCP_ARGS=["-m", "splunk_mcp_server"]

# Auth
RUNBOOKMIND_AUTH_ENABLED=true
RUNBOOKMIND_DEFAULT_ADMIN_API_KEY=your-admin-key-here

# Vault
RUNBOOKMIND_VAULT_ENABLED=true
RUNBOOKMIND_VAULT_MASTER_KEY=your-32-byte-hex-master-key

# Splunk direct credentials (for query runner)
SPLUNK_HOST=localhost
SPLUNK_PORT=8089
SPLUNK_USERNAME=admin
SPLUNK_PASSWORD=your_password
```

### Step 2: Swap the Dependency Injections in `dependencies.py`

**Schema Discovery:**
```python
# FROM:
from app.schema.provider import MockSchemaProvider
schema_provider = MockSchemaProvider()

# TO:
from app.schema.discovery import SchemaDiscoveryEngine
schema_provider = SchemaDiscoveryEngine()
```

**SPL Generator/Optimizer/Explainer:**
```python
# FROM (in RunbookService or wherever compiler is instantiated):
from app.spl.generator import MockGenerator
from app.spl.optimizer import MockOptimizer
from app.spl.explainer import MockExplainer

# TO:
from app.splunk.adapters.mcp_generator import SplunkMCPGenerator
from app.splunk.adapters.mcp_optimizer import SplunkMCPOptimizer
from app.splunk.adapters.mcp_explainer import SplunkMCPExplainer
```

**Query Runner:**
```python
# FROM:
_query_runner_instance = MockQueryRunner()

# TO:
from app.runtime.runner import SplunkQueryRunner  # may need to create this
_query_runner_instance = SplunkQueryRunner(
    host=os.environ["SPLUNK_HOST"],
    port=int(os.environ["SPLUNK_PORT"]),
    username=os.environ["SPLUNK_USERNAME"],
    password=os.environ["SPLUNK_PASSWORD"]
)
```

### Step 3: Install the Splunk MCP Server

```bash
pip install splunk-sdk
pip install splunk-mcp-server   # verify exact package name from Splunk developer docs
```

### Step 4: Enable Auth + Vault

```bash
RUNBOOKMIND_AUTH_ENABLED=true
RUNBOOKMIND_VAULT_ENABLED=true
```

### Step 5: Connector Real Credentials

For each connector (Slack, Jira, etc.), use **real tokens** when calling the connector creation API — not `mock-` prefixed ones. The real HTTP implementation is already there.
